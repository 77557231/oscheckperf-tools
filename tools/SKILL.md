# 系统基准测试工具

## ️ 核心概念与架构认知

- **项目定位**：理解该工具是为 Vastbase、openGauss 和 PostgreSQL 设计的系统级性能基准测试工具，用于评估服务器硬件性能边界。
- **目录结构认知**：
    - `oscheckperf/`：根目录。
    - `oscheckperf`：主入口脚本（单文件集成）。
    - `output/`：测试结果输出目录。
    - `tools/`：辅助工具目录。
    - `README.md` / `README.en.md`：中英文文档。
- **设计原则**：避免使用 `lib` 等复杂依赖目录，保持代码简洁，所有功能集成在单一脚本中。

##  Shell 脚本编程规范

### 代码结构与风格
- **模块化设计**：必须使用函数封装功能模块，提高可维护性。
- **命名规范**：函数名必须使用下划线分隔的小写字母（例如：`run_cpu_test`）。
- **注释规范**：所有代码注释必须使用英文，清晰描述逻辑。
- **语言分离**：代码逻辑用英文注释，文档（README, skill.md）用中文编写。

### 命令行参数处理
- **参数格式**：支持短选项（如 `-c`, `-o`）和 `KEY=VALUE` 格式的参数覆盖。
- **优先级逻辑**：确保命令行参数的优先级高于配置文件中的设置。
- **变量命名**：参数名统一使用大写字母和下划线（例如：`DURATION`, `CPU_MAX_PRIME`）。

### 通用开发实践
- **重复性工作处理**：
    - **参数管理**：统一通过配置文件和命令行解析，严禁硬编码。
    - **错误处理**：建立统一的错误捕获与处理机制，增强脚本稳定性。
    - **日志输出**：标准化日志格式，便于调试。

##  报告生成与数据管理规范

### 结果输出分类
必须严格区分三种类型的输出文件，以满足不同场景需求：

1. **命令结果输出 (`data_*_all_results.log`)**
    - **目标**：实时反馈，快速查看关键指标。
    - **内容**：单个 IP 下的关键指标，简洁明了。
    - **场景**：实时监控进度。
2. **汇总报告 (`report_benchmark_*.log`)**
    - **目标**：汇总分析，横向对比。
    - **内容**：多 IP 相同指标对比，表格化呈现。
    - **场景**：技术人员综合分析性能差异。
3. **原始数据 (`original_data_*_all_results.log`)**
    - **目标**：归档留存，深度挖掘。
    - **内容**：完整的原始测试数据，无加工。
    - **场景**：自定义数据处理或深入分析。

### 报告生成要求
- 生成结构化的测试报告，包含系统信息和各模块结果。
- 结果文件统一存储在 `output` 目录。
- 格式统一，便于脚本解析和后续工具对比。

##  文档维护与版本控制

### 文档同步机制 (关键)
每次代码更新（新功能或 Bug 修复）必须同步执行以下操作：
- **更新 `README.md`**：同步变更内容，保持结构统一。
- **更新 `README.en.md`**：内容与中文版保持一致，仅语言不同。
- **一致性检查**：确保帮助信息与文档描述一致。

### 标签与版本管理
- **版本号规则**：Git Tag 与版本号必须一致（如 `v0.6.2`）。
- **Tag 描述格式**：必须遵循特定格式记录变更。

> **格式模板：**
> ```text
> vX.X.X - 简短描述
> 1. 变更点1
> 2. 变更点2
> 验证结果：...
> ```

- **示例**：
    ```text
    v0.6.2 - 参数体系优化与功能增强
    1. 参数体系优化：统一IO测试参数命名，区分通用参数(IO_)、SYSBENCH专用参数(SYSBENCH_)和FIO专用参数(FIO_*)
    2. 功能增强：支持SYSBENCH_PROFILES和FIO_PROFILES多值参数，可同时执行多个测试模式
    3. 代码简化：移除IO_TEST_PATH向后兼容性代码，优化临时文件管理
    验证结果：所有测试正常执行，远程参数传递正确，日志包含执行命令，报告生成正确
    ```

##  最佳实践与流程

### 标准开发流程
1. **需求分析**：明确测试场景和功能点。
2. **代码实现**：遵循 Shell 规范，函数化开发。
3. **测试验证**：确保功能正常，结果准确。
4. **文档更新**：同步更新中英文 README。
5. **版本发布**：打 Tag 并填写规范描述。

### 质量保障
- **环境一致性**：测试时确保环境稳定，排除外部干扰。
- **时长控制**：测试时长需合理，以保证结果统计学上的稳定性。
- **依赖管理**：定期更新依赖工具的版本。
